# N07 API接口需求文档设计规格

## 目标

为 `document-filler` 新增材料类型 `01-API接口开发_需求文档`。生成器读取台账中 `服务目录=N07-API接口开发` 的工单及接口数据，从 `自测报告附件` 的内嵌 Word 文档解析接口输入、输出参数，填充指定的 `01-需求文档.docx` 模板。

命令中的输出参数为目录时，本材料的默认文件名固定为 `01-需求文档.docx`。本次真实输出位置为：

`C:\Users\p\Desktop\新调整\结果\01-需求文档.docx`

## 范围

本次实现包括：

- 支持合并单元格台账，按工单号聚合工单信息和接口清单。
- 生成业务场景、需求清单、工单需求口径和 API 接口清单。
- 从 `自测报告附件` 内嵌 DOCX 的“测试内容”章节解析接口输入、输出参数。
- 保留模板的封面、修订记录、共享开放方案、图片、页眉页脚和既有样式。
- 更新目录域并验证生成的 Word 文档。
- 为后续 N07 材料提供轻量注册表和可复用的台账、附件、Word 章节工具。

本次不包括：

- 不迁移或重写现有 N08 材料生成器。
- 不为尚未提出的 N07 设计文档、测试报告或上线记录预先实现业务逻辑。
- 不调用在线大模型服务生成正文。
- 不覆盖原始模板。

## 架构

采用“材料注册表 + 共享能力 + 独立材料生成器”的渐进式结构：

```text
scripts/
  fill_document.py
  materials/
    __init__.py
    registry.py
    shared/
      __init__.py
      ledger.py
      embedded_docx.py
      word_sections.py
    n07/
      __init__.py
      api_requirement.py
tests/
  test_api_requirement.py
```

### `scripts/materials/registry.py`

保存新式材料的声明信息：

- 材料类型名称
- 默认输出文件名
- 生成器入口
- 允许的输出扩展名

首个注册项：

```python
"01-API接口开发_需求文档" -> {
    "default_filename": "01-需求文档.docx",
    "builder": build_api_requirement_document,
}
```

`fill_document.py` 在现有分支之前查询注册表。找到新式材料时调用注册生成器；未找到时继续进入现有分支。这样不会改变旧材料的行为。

### `scripts/materials/shared/ledger.py`

负责台账结构，不包含任何具体材料的正文规则：

- 动态读取表头。
- 将合并区域左上角的值映射到区域内各行。
- 按 `服务目录` 筛选。
- 按 `工单号` 聚合，同一工单只保留一份需求单号、标题、描述和程序数。
- 收集每一行的 `结果表清单`，解析为接口中文名和接口英文名。
- 记录工单覆盖的原始行范围，供附件锚点匹配。

程序数按工单聚合后累计，不能按合并区域展开后的接口行重复相加。当前真实台账第 2-4 行属于同一工单，因此总数为 3，不是 9。

### `scripts/materials/shared/embedded_docx.py`

负责从 Excel 包中提取锚定于指定列的 OLE Word 附件：

- 解析 worksheet、relationship 和 VML anchor。
- 只接受锚定在 `自测报告附件` 列的对象。
- 根据锚点所在行或合并区域关联工单。
- 从 OLE `Package` 或 `Ole10Native` 流读取 DOCX 载荷。
- 返回工单号到临时 DOCX 文件的映射。

该模块不解析业务章节。以后 N07 测试或设计材料需要读取内嵌 Word 时可以复用。

### `scripts/materials/shared/word_sections.py`

提供模板级操作：

- 按标题样式和标题文字定位章节边界。
- 替换两个章节标题之间的段落与表格。
- 复制模板段落样式、表格边框、底纹和单元格格式。
- 生成自适应列数的 Word 表格。
- 保留未被替换的 XML 节点、图片关系、页眉页脚和节设置。

该模块不包含“业务场景”“需求口径”等业务文字。

### `scripts/materials/n07/api_requirement.py`

只实现 `01-API接口开发_需求文档` 的数据转换和章节填充规则。它依赖三个共享模块，但不被其他材料生成器反向依赖。

## 数据模型

生成器使用以下概念模型：

```text
ApiWorkOrder
  demand_no
  work_order_no
  title
  description
  program_count
  source_rows
  interfaces[]
  self_report_path

ApiInterface
  chinese_name
  english_name
  input_groups[]
  output_groups[]
  purpose

ParameterGroup
  label
  headers[]
  rows[][]
```

`ParameterGroup` 保留附件原表的列结构。输入参数可能包含“请求头”“请求体”“IdAuthData”等多个分组，不压缩成固定三列；输出参数同样保留实际列数和备注。

## 数据流

1. 注册表解析材料类型和输出路径。
2. 台账模块读取 `N07-API接口开发`，还原合并行并生成工单列表。
3. 每个 `结果表清单` 按最后一个空格拆分为接口中文名和英文名。
4. 附件模块从 `自测报告附件` 列提取每个工单的内嵌 DOCX。
5. API 生成器读取附件“测试内容”章节，按接口中文名精确匹配接口段落。
6. 在接口段落中，以“输入参数”和“输出参数”为边界收集标签和表格。
7. 根据工单描述、接口名称、主要输入和输出字段生成接口作用描述。
8. 填充模板章节并保存到输出路径。
9. 通过 Word COM 更新目录域。

## 模板填充规则

### 业务场景

保留模板“业务场景”标题后的第一段总体说明。替换该章节末段，内容包括：

- 所有去重后的工单标题。
- 所有工单 `程序数` 的合计。
- 明确写为提供多少个 API 接口服务。

当前真实数据生成的核心信息为：

`在本服务周期内，围绕市公安局-出入境证件身份认证-共享接口开展API接口开发，共提供3个API接口服务。`

### 需求说明和需求清单

需求清单说明格式：

`服务周期内，共有{需求单数}张需求单，{工单数}张工单涉及{接口总数}个API接口服务。具体需求单、工单和产出如下表：`

表格列沿用模板：

| 序号 | 对应需求单编号 | 对应工单编号 | 工单内容 | 涉及共享任务数量（个） |
|------|----------------|--------------|----------|--------------------------|

每个工单一行，末行显示合计。合并行展开后不能产生重复工单行。

需求清单表格之后、`共享开放方案` 之前，按工单生成：

- Heading 3：`{工单号}_{工单标题}`
- 正文：`需求口径：{工单描述}`

### 共享开放方案

整节保留模板原文和图片，不参与动态替换。

### API接口清单

删除模板中的示例接口内容，按台账接口顺序生成：

- Heading 3：接口中文名。
- 正文：接口作用描述。
- 正文：`计划供数方式：API接口对外服务`。
- 正文：`计划更新频率：无`。
- 正文：`接口输入参数：`，随后插入一个或多个参数分组表格。
- 正文：`接口输出参数：`，随后插入一个或多个参数分组表格。

英文接口名用于附件匹配的辅助校验，不写入标题。

## 接口作用描述

作用描述只使用可验证信息：工单描述、接口中文名、主要输入字段、主要输出字段以及接口在同一工单中的先后关系。

生成规则按接口语义变化：

- 令牌类接口说明凭证获取和后续调用关系。
- 申请类接口说明提交认证上下文并返回业务流水号、随机数等申请结果。
- 请求或查询类接口说明接收身份或查询条件并返回核验、查询结果。
- 其他接口根据输入和输出字段概括实际数据流。

文风遵循 `humanizer-zh`：直接写功能，不使用“赋能、彰显、至关重要、确保、重要支撑”等宣传或套话，不机械使用三段式结构，不添加泛化总结。

## 匹配和错误处理

以下问题直接终止生成并列出工单号或接口名：

- 台账缺少必要列。
- 指定服务目录没有数据。
- 工单没有可解析的 `自测报告附件`。
- 附件不是可识别的 DOCX 包。
- 接口中文名无法在附件“测试内容”章节中匹配。
- 接口缺少输入或输出参数表。
- 模板缺少业务场景、需求清单、共享开放方案或 API 接口清单章节。

错误发生时不输出带有“待补充”占位符的验收文档。输出目录不存在时自动创建。

## 输出路径

命令示例：

```powershell
python scripts/fill_document.py `
  --service-dir "N07-API接口开发" `
  --material-type "01-API接口开发_需求文档" `
  --excel "C:\Users\p\Desktop\新调整\验收台账清单（更新）.xlsx" `
  --template "C:\Users\p\Desktop\新调整\N07 数据共享、开发、授权订阅服务\01-API接口开发\01-需求文档.docx" `
  --output "C:\Users\p\Desktop\新调整\结果"
```

输出文件：

`C:\Users\p\Desktop\新调整\结果\01-需求文档.docx`

## 测试策略

自动化测试包括：

- 合并单元格继承和按工单聚合。
- 程序数按工单只累计一次。
- 一个工单多行接口的顺序和名称解析。
- OLE Package 附件提取和锚点关联。
- “测试内容”章节的接口中文名匹配。
- 多组输入参数及嵌套对象表格解析。
- 输出参数表解析。
- 缺少附件、接口或参数表时的错误消息。
- 材料注册、默认输出文件名和主入口分发。
- 模板静态章节、图片和样式保留。
- 真实台账和真实模板端到端生成。

最终验证包括：

- 运行全部单元测试。
- 打开生成的 DOCX 检查章节、表格数量和关键文本。
- 通过 Word COM 转换为 PDF。
- 将 PDF 页面渲染为图片，检查表格溢出、空白页、标题孤行和内容重叠。

## 后续新增 N07 材料

新增材料时遵循以下步骤：

1. 在 `scripts/materials/n07/` 新建独立生成器。
2. 只复用所需的共享模块。
3. 在注册表增加材料类型、默认文件名和入口。
4. 为该材料单独增加测试文件或测试类。
5. 不修改其他材料生成器的业务规则。

只有当第二种材料证明存在稳定重复逻辑时，才继续扩展共享模块；不为假设中的未来需求预建抽象层。
