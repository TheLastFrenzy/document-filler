# 文档填充规则详细说明

## 台账清单列结构（动态读取，不硬编码列数）

Excel 第一行为列名，脚本动态读取所有列。筛选仅按 `服务目录` 列进行。

新版台账标准列名：

| 新列名 | 兼容旧列名 | 脚本内部标准字段 |
|--------|------------|------------------|
| `工单标题` | `工单内容` | `工单内容` |
| `程序数` | `报表统计次数` | `报表统计次数` |
| `工单描述` | `业务说明`、`业务描述` | `业务说明` / `业务描述` |
| `结果表清单` | `统计分析结果表清单` | `统计分析结果表清单` |

脚本读取台账时会将新版列名映射为内部标准字段；旧版台账仍可兼容。新材料和新模板说明优先使用新版列名。

## 01-API接口开发_需求文档

### 输入与输出

- 服务目录固定使用 `N07-API接口开发`。
- 模板为 Word `.docx`，必须包含 `业务场景`、`需求说明`、`需求清单`、`共享开放方案` 和 `API接口清单` 标题。
- 输出参数为文件夹时，默认文件名为 `01-需求文档.docx`；输出文件夹不存在时自动创建。
- 原模板不被覆盖。

### 台账列

| 列名 | 用途 |
|------|------|
| `服务目录` | 只筛选 `N07-API接口开发` |
| `需求单号` | 需求清单中的对应需求单编号 |
| `工单号` | 工单聚合键和需求内容标题前缀 |
| `工单标题` | 业务场景、需求清单和工单标题 |
| `程序数` | 按工单累计 API 接口服务总数 |
| `工单描述` | `需求口径` 正文和接口作用描述的业务依据 |
| `结果表清单` | 每行一个 `接口中文名 + 空格 + 接口英文名` |
| `自测报告附件` | 每个工单内嵌一个 Word 自测报告附件 |

台账允许合并行。生成器将合并区域左上角的值继承到区域内各行，再按 `工单号` 聚合。`程序数` 每个工单只累计一次，不按接口行重复相加。接口顺序按台账原始行顺序保留。

### 业务场景

保留模板 `业务场景` 下的第一段总体说明，替换该章节末段。新段落罗列所有去重后的工单标题，并写明 `程序数` 合计：

`在本服务周期内，围绕{工单标题列表}开展API接口开发，共提供{程序数合计}个API接口服务。`

### 需求说明和需求清单

需求清单说明文字：

`服务周期内，共有{去重需求单数}张需求单，{去重工单数}张工单涉及{程序数合计}个API接口服务。具体需求单、工单和产出如下表：`

表格列：

| 序号 | 对应需求单编号 | 对应工单编号 | 工单内容 | 涉及共享任务数量（个） |
|------|----------------|--------------|----------|--------------------------|

每个工单一行，末行显示总计。表格之后、`共享开放方案` 之前，按工单生成：

```text
Heading 3: {工单号}_{工单标题}
Normal: 需求口径：{工单描述}
```

- 需求清单说明段落复制模板原段落的 `pPr` 和首个文本 run 的 `rPr`，不重新指定通用 Normal 样式。
- 需求清单表复制模板的表格属性、列宽、表头行、数据行和总计行原型；总计行保留模板前两列合并方式。
- 每个工单标题复制模板首个工单标题的 Heading 3 直接格式和 Word 编号定义，共用同一编号列表，按 `1、2、3、...` 自动递增；标题文本本身不硬编码序号。
- 需求口径复制模板示例正文的宋体、字号、首行缩进和行距等直接格式。

`共享开放方案` 整节保留模板原文和图片。

### 自测报告解析

生成器从 Excel worksheet relationship、VML anchor 和 OLE embedding 定位 `自测报告附件` 列中的内嵌 Word。附件按锚点所在行或合并区域关联工单。

在附件 `测试内容` 章节中按接口中文名精确匹配。每个接口段落内：

- `输入参数` 到 `输出参数` 之间的表格属于输入参数。
- `输出参数` 到本接口测试结果或下一接口之间的表格属于输出参数。
- `请求头`、`请求体`、`userInfo`、`IdAuthData` 等表格前标签作为参数分组名保留。
- 表头、数据行、原始列数和原始列宽比例全部保留；列宽按模板页面宽度等比缩放。
- 输入和输出表格分别复制模板对应输入表、输出表的表格、表头行和数据行原型；仅按自测报告的原始列数和列宽比例调整网格。
- 表头居中并使用模板灰色底纹，数据单元格保持模板左对齐和字体格式，长表跨页时重复表头。

### API接口清单

删除模板 `API接口清单` 下的全部示例接口，按台账顺序生成：

```text
Heading 3: {接口中文名}
Normal: {接口作用描述}
Normal: 计划供数方式：API接口对外服务
Normal: 计划更新频率：无
Normal: 接口输入参数：
[一个或多个输入参数分组和表格]
Normal: 接口输出参数：
[一个或多个输出参数分组和表格]
```

API 标题、接口说明、计划供数方式、计划更新频率、输入/输出参数标签均复制模板对应示例段落的直接格式。API 标题沿用模板 Word 自动编号并按接口顺序递增；计划供数方式和计划更新频率保留模板项目符号。

接口作用描述结合工单标题、工单描述、接口名称、主要输入字段和主要输出字段生成：先概括工单背景，再说明本接口接收/查询的输入、返回的输出和用途。输入/输出字段摘要优先使用参数表中的 `名称`、`参数说明`、`字段中文名`、`字段注释`、`备注` 等中文说明列；只有缺少中文说明时才回退到英文参数名。令牌接口说明凭证获取和后续调用关系，申请接口说明认证申请与业务流水号，请求或查询接口说明核验条件和返回结果。文字直接说明数据流和用途，不使用 `赋能`、`彰显`、`至关重要`、`确保`、`重要支撑` 等套话。

### 严格错误

以下情况停止生成，并在错误中列出具体工单或接口：

- 台账缺少必要列或没有匹配服务目录的数据。
- 工单没有可解析的 `自测报告附件`。
- 附件不是 DOCX 包。
- 接口中文名无法在自测报告中匹配。
- 接口缺少输入或输出参数表。
- 模板缺少必要章节。

错误时不生成带“待补充”占位内容的验收文档。

### 4.2 接口出入参配置表

删除模板 `接口出入参配置表` 下、`接口组件配置表` 前的示例工单和接口，按台账顺序生成：

```text
Heading 3: {工单标题}
  Heading 4: {接口中文名}
    Heading 5: 接口入参配置表
    [一个或多个入参分组标签和表格]
    Heading 5: 接口出参配置表
    [一个或多个出参分组标签和表格]
```

- `Heading 3`、`Heading 4`、`Heading 5` 分别复制模板 4.2 节对应标题原型，保留标题层级、字体、字号、缩进和目录识别能力。
- 输入参数表复制模板首个入参表格原型，输出参数表复制模板首个出参表格原型；按自测报告实际表头和列数调整表格网格，保留模板边框、底纹、单元格字体和长表跨页表头。
- 参数分组名使用模板正文段落原型插入，不作为目录标题。
- 未提及的模板静态章节（如 `接口定义表`、`接口组件配置表`、`API数据模型开发`）直接保留。
- 生成后通过 Word COM 更新目录域。

    Body Text: {交付要求内容}  (首行缩进480twips)

#### 数据需求与交付要求规范化

- `数据需求`、`交付要求` 为空或仍是固定模板式描述时，脚本会结合 `工单内容`、`业务说明`、`02-数据报表_设计文档-数据来源库表清单对应数据目录代码` 推理补齐。
- 如果命令提供 `--catalog` 数据目录数据路径，01 需求文档会进一步读取 `关联资源信息` 和 `数据项`，将目录代码对应的资源名称、资源编码、字段中文名纳入 `数据需求` 和 `交付要求`，避免只按目录编码套固定句式。
- 推理生成时会优先使用已有证据：工单主题、业务说明中的报表清单、数据目录资源名称、数据项字段名，以及台账中可解析的程序/字段信息。
- `数据需求` 会说明本次需求基于哪些目录/资源、围绕什么业务场景、需要梳理哪些关键字段、统计范围和结果口径，并指出数据内容需与业务说明及交付附件中的报表范围保持一致。
- `数据需求` 和 `交付要求` 不使用固定开头/固定结尾模板；生成时按工单内容、工单号、业务说明和目录代码稳定组合正文句，不额外追加“归档、追溯、后续核对”等通用收尾套话。
- `交付要求` 会说明交付材料需包含报表成果、数据目录清单、字段口径说明和必要附件；交付物形式按台账 `交付物` 附件或文本中的后缀识别，只有存在 `.xlsx/.xls/.docx/.doc/.pdf/.zip` 等证据时才写明 Excel、Word、PDF 或压缩包等格式，不凭空预设所有工单都同时交付 Excel 和 Word。
- 正文只描述交付成果本身，例如 `Excel电子表格报表`、`Word文档说明材料`、`PDF版定稿或签收材料`；不得写入 `交付物后缀显示`、`以Word格式提交`、`以Excel格式提交` 这类生成逻辑痕迹。
- `交付要求` 中的文件命名要求应结合实际工单表达，通常对应工单编号、统计时间段和报表主题；涉及接口类材料时再补充接口编号或版本线索。
- `交付要求` 会要求验收时对照资源名称、目录代码、关键字段、程序处理结果和业务说明确认一致性，不额外补写泛化的归档或后续管理结尾。
- `数据需求` 和 `交付要求` 应减少“确保、彰显、关键、保持一致”等泛化表达，直接写清数据来源、字段、统计范围、交付物、命名和归档要求。
- `数据需求` 引用数据目录代码但只列举部分目录时，必须带“等目录”，例如 `002420412/000269、002420412/000114等目录`。
- `交付要求` 不使用“检查”“抽查”“复核”等措辞，改用“确认”“对照”等直接表达。
- 当台账中存在数据目录代码列时，`交付要求` 优先使用该列目录代码；附件或正文中的长数字编号不作为目录代码写入。
- `交付要求` 会移除 `【】`、`[]` 这类附件解析残片。

#### 内嵌Excel清单表格触发条件
在"业务说明"字段中检测模式：
- `本次工作拟产出以下N份报表成果` （带"以下"）
- `本次工作拟产出N份报表成果`    （不带"以下"）

3. 紧随其后插入 2 列表格（序号 + 名称），每行一个报表名称

### 核心列（按列名匹配）

- `工单内容`：第3节每个小节的标题(Heading 2)
- `内容描述`：业务分析表格的内容描述

- `交付物`：工单级附件列；同一工单多行合并时，脚本按整组行提取该列 OLE 附件文件名

`数据内容` 为空或仍是固定模板式描述时，脚本会结合工单描述、业务说明和数据目录代码推理生成更贴近该工单的说明。

新版 N08 数据报表台账缺少旧版 `内容描述`、`业务场景`、`结果形式`、`数据处理逻辑` 列时，脚本按以下规则补齐，避免 02 设计文档出现空白业务分析项：

- `内容描述`：优先结合 `业务说明`、`业务描述`、`工单内容` 和数据目录代码生成。
- `业务场景`：优先结合 `业务描述`、`业务说明`、`工单内容` 和数据目录代码生成；生成正文应描述业务方为什么需要该报表、用于哪个专项工作或核验场景，不写 `上线后用于` 这类固定尾句。
- `数据内容`：生成正文应描述数据覆盖的目录/资源、关键字段、统计对象和结果口径；涉及最终形成的报表或材料名称时，优先使用交付附件解析出的交付物名称，不得用程序中文名替代交付物名称；不得写 `本节围绕`、`附件中可见`、`展示附件` 等文档生成痕迹。
- `结果形式`：优先结合 `交付物` 解析出的附件名称；附件为 OLE Package 时优先读取 `Ole10Native` 原始文件名，其次读取压缩包内文件名，再结合 Office 文档属性或 Excel sheet 名推断；仍无法取得可读名称时使用工单标题作为交付文件名，不得回退为程序中文名。
- `数据处理逻辑`：优先结合数据目录代码、`统计分析结果表清单`、`程序XML文本` 解析出的程序/字段信息生成；若该列本身是 `DISPIMG` 图片引用，则保留图片逻辑；文字描述不得出现“清洗”“抽取”“加密”“质量检查”等字眼。
- `deliverable_rowXX`、`package`、`Workbook`、`Ole10Native_embedded` 等附件提取内部兜底名不得写入文档；遇到这类名称且无法恢复真实附件名时，改用工单标题作为交付文件名，不写入“本次交付材料/本次交付报表”等泛化占位。

### 额外参数

- `--catalog`：数据目录数据Excel路径（必填），包含两个Sheet：
  - "关联资源信息"：根据数据目录代码匹配资源名称和资源编码

- 台账列 `自测报告附件`（兼容旧列名 `03-数据统计分析_测试文档_工单自测报告附件`）：每个工单内嵌一个 Word 自测报告附件，附件覆盖该工单所有统计分析结果表。

### 自测报告解析

从台账 Excel 的 OLE `package` 流提取内嵌 Word 自测报告，按所在行或合并区域关联工单。自测报告中读取 `4.检查和测试内容` 章节，按程序英文名匹配以下七类截图：

1. 程序中英文名称规范性
2. 表命名规范性
3. 表字段名规范性
4. 表及表字段注释规范性
5. 程序逻辑检查
6. 程序代码注释规范性
7. 程序运行测试

若台账英文名和自测报告英文名只有后缀差异或相似度高，允许近似匹配并在程序标题下写明匹配说明。

### Word 生成规则

- 模板内容保留：直接保留模板封面、修订记录、目录和 1-4 章静态内容。
- 目录更新：生成后通过 Word COM 更新模板中的目录域，目录应指向实际生成的 5.x 程序级条目和 `6.测试结论`。
- `5.测试内容`：替换模板示例内容，按台账中每个统计分析结果表生成一节，标题使用结果表中文名。
- 每个程序下生成七个小节，截图按自测报告中出现顺序插入；同一小节多张图时依次插入。
- 如果某工单没有找到自测报告附件，仍保留该工单所有程序和七个小节，内容标注“截图待补充”。
- `6.测试结论`：正文使用模板原版结论描述；下方表格只更新总数：

| 测试内容 | 测试项 | 通过项 | 通过率 |
|----------|--------|--------|--------|
| 数据测试 | 程序总数 | 程序总数 | 100% |
| 程序测试 | 程序总数 | 程序总数 | 100% |

## 04-数据统计分析_结果表及使用说明

此材料输出为 Excel `.xlsx` 工作簿，不是 Word 文档。入口仍使用 `scripts/fill_document.py`，材料类型为 `04-数据统计分析_结果表及使用说明`，并且必须提供 `--catalog` 数据目录数据路径。

### 核心列（按列名匹配）

- `服务目录`：筛选值通常为 `N08-数据统计分析`
- `程序XML文本`：draw.io/mxGraphModel 形式的程序流转 XML，内含 SQL、表达式、节点标签等
- `结果表清单`：结果表中文名 + 英文表名

### 额外参数

- `--catalog`：数据目录数据Excel路径（必填），用于补齐源表/结果表资源信息和字段信息。
- `--template`：Excel模板 `.xlsx`，应包含以下 sheet：
  - `说明`
  - `1、数据源表list`
  - `2、表融合关系`
  - `3、数据统计分析结果表list`
  - `4、数据统计分析结果表详情`

### 工作簿生成规则

1. 按 `服务目录` 从台账清单筛选记录。
2. 对每条记录解析 `程序XML文本`：
   - 从 SQL 中识别结果表、源表、关联表和 CTE。
   - 排除结果表自身、`DUAL`、`ALL_TABLES` 等非业务源。
   - 从 create/insert/select 片段提取结果表字段；如 XML 无字段，则回退到数据目录数据字段。
3. 结合数据目录数据补齐：
   - 源表/结果表的资源名称、资源编码、资源类型、单位名称、数据目录代码。
   - 数据项名称、英文名称、数据类型、字段描述。
4. 全工作簿字体统一为 `宋体`。
5. 需要加深背景的标题、表头和标签单元格统一使用 `白色,背景1,5%` 的浅灰填充。

### Sheet 1：`1、数据源表list`

按解析出的源表去重后生成源表清单：

| 资源名称 | 数据提供方 | 资源类型 | 数据源表的资源信息（表名） |
|----------|------------|----------|----------------------------|

`数据提供方` 统一填 `上海市大数据中心`；`资源类型` 统一填 `库表`。

### Sheet 2：`2、表融合关系`

每个结果表固定生成 3 行：

1. 标题行：仅在 A 列写入 `2.{序号} {结果表英文名}`，不合并 A:C，不设置背景色填充。
2. `文字描述` 行：B列写标签、C列写业务化分点总结，标签单元格不设置背景色填充。
3. `数据处理流程图` 行：B列写标签、C列内嵌流程图图片，标签单元格不设置背景色填充。

相邻结果表程序块之间保留一个空行，空行行高统一设置为 `20`。

#### 文字描述要求

- 不能直接复制 XML 文本、SQL 片段、节点标题或注释。
- 必须分析 XML/SQL 后，用业务可读语言分点总结结果表的数据梳理逻辑。
- 业务逻辑表述不得使用“清洗”“抽取”“加密”“质量检查”等程序化或不适合验收材料的字眼；可改用“范围确认”“字段口径整理”“口径核对”“结果形成”等业务化说法。
- 推荐结构：
  1. 数据来源准备：说明读取哪些源表作为基础数据。
  2. 数据范围确认：总结有效记录、分区批次、状态、区域、服务范围等筛选口径。
  3. 融合加工计算：总结 join、union、group by、count、case、环比/同比/占比等加工逻辑。
  4. 结果字段整理：列出主要输出字段和运行信息。
  5. 结果输出：说明写入哪个结果表，供数据统计分析应用使用。

#### 数据处理流程图要求

- 主要反馈源表到结果表的血缘关系，不展示 XML 程序步骤细节。
- 样式参考模板：黑色背景、浅橙色节点、深灰色箭头。
- 结构为：`开始` -> 一个或多个源表节点 -> `统计融合为 {结果表中文名/英文名}` -> `推送至应用开发部` -> `结束`。
- 多源表时使用并行源表节点汇聚到结果表；源表过多时只展示主要源表并用“其他N张源表”归并。
- 源表节点优先显示资源名称 + 英文表名；英文长表名按下划线分组换行，避免单词被拆得不可读。

### Sheet 3：`3、数据统计分析结果表list`

每个结果表生成一行：

| 序号 | 资源编目（非必填） | 资源名称 | 数据提供方 | 资源类型 | 数据统计分析表的资源信息（表名） |
|------|--------------------|----------|------------|----------|----------------------------------|

- `序号` 按结果表顺序自增。
- `资源编目（非必填）` 用结果表英文名匹配 `数据目录数据.xlsx` 的 `关联资源信息.资源编码`，填写匹配记录的 `数据目录代码`。
- `资源名称` 使用台账 `结果表清单` 解析出的中文名。
- `数据提供方` 使用匹配到的 `单位名称`；未匹配或为空时填 `上海市大数据中心`。
- `资源类型` 使用匹配到的 `资源类型`；未匹配或为空时填 `库表`。
- `数据统计分析表的资源信息（表名）` 使用台账 `结果表清单` 解析出的英文表名。

### Sheet 4：`4、数据统计分析结果表详情`

每个结果表生成一个字段详情区块：

1. 标题行：仅在 A 列写入 `4.{序号} {结果表中文名}`，不合并 A:H，不设置背景色填充。
2. 结果表英文名行：仅在 B 列写入 `{结果表英文名}`，不合并 B:H，居中加粗。

| 字段中文名 | 字段英文名 | 字段类型 | 默认 | 不可为空 | 唯一 | 字段注释 |
|------------|------------|----------|------|----------|------|----------|

字段优先来自 XML 中的结果表建表/插入语句，XML 不完整时使用数据目录数据补齐。字段描述为空时填 `No`。

### 验证要求

- 生成的 `.xlsx` 必须能被 Excel 打开。
- `2、表融合关系` 的结果表段数应等于筛选出的结果表数量。
- `2、表融合关系` 的内嵌图片数量应等于结果表数量。
- 扫描 `文字描述`，不得出现 `mxGraphModel`、`<mxCell`、`create table`、`insert into`、`truncate table`、`sql语句` 等原始 XML/SQL 痕迹。

## 03-数据报表_上线记录

### 核心列（按列名匹配）

- `需求单号`：对应需求单编号
- `工单号`：对应工单编号
- `工单内容`：各小节标题(Heading 1)，格式为"{工单内容}的上线记录"
- `报表统计次数`：数值，用于统计和表格
- `需求描述`：产出说明章节的需求描述内容；新版台账缺少该列或为空时，依次回退到 `业务说明`、`业务描述`、`工单内容`
- `上线交付截图1`、`上线交付截图2`：上线交付截图章节的内嵌图片（DISPIMG 公式引用）
- `使用记录截图1`、`使用记录截图2`：使用记录章节的内嵌图片（DISPIMG 公式引用）

### 需求来源填充规则

#### 描述文字
格式：`服务周期内，共有{N}张需求单，{M}张工单涉及{K}次数据报表服务。具体需求单、工单和产出如下表：`
- N = 去重后需求单号数量
- M = 去重后工单号数量
- K = 报表统计次数之和

#### 表格（5列）
| 序号 | 对应需求单编号 | 对应工单编号 | 工单内容 | 报表统计次数 |
|------|---------------|-------------|---------|-------------|
| 自增 | 需求单号 | 工单号 | 工单内容 | 报表统计次数 |
| 合计(合并前2列) | - | - | - | K |

表格使用实线边框（single, sz=4）。

### 各工单章节填充规则

每条筛选记录生成以下结构：
```
Heading 1: {工单内容}的上线记录
  Heading 2: 产出说明
    Body Text: 需求编号：{需求单号}    对应工单编号：{工单号}  (首行缩进480twips，允许西文在单词中间换行)
    Body Text: 需求描述：{需求描述内容}  (首行缩进480twips)
    Normal: 统计报表：{报表统计次数}次。  (首行缩进480twips，允许西文在单词中间换行)
  Heading 2: 上线交付截图
    Body Text: [按实际存在的上线交付截图列内嵌图片；缺失时不额外保留空白占位]
  Heading 2: 使用记录
    Body Text: 使用记录如下：
    Body Text: [按实际存在的使用记录截图列内嵌图片；缺失时不额外保留空白占位]
```

### 图片提取

- 图片从台账清单的 `xl/cellimages.xml` 中提取（WPS 格式）
- DISPIMG 公式中的 ID 与 cellimages.xml 中的 image name 匹配
- 图片路径通过 `xl/_rels/cellimages.xml.rels` 映射到 `xl/media/` 下的实际文件

### 格式要求

- 表格：实线边框（single, sz=4），表头加粗+背景色 F1F1F1
- Body Text 段落：首行缩进 `firstLine=480`
- 产出说明首行格式固定为 `需求编号：{需求单号}\t对应工单编号：{工单号}`，标签后不额外插入制表符
- 产出说明中的 `统计报表：{报表统计次数}次。` 段落同样设置首行缩进 `firstLine=480`
- 需求编号行和统计报表次数行设置 `w:wordWrap`，允许西文在单词中间换行
- TOC：通过 PowerShell COM 调用 Word 自动更新，锚定"需求来源"段落前
